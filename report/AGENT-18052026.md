## 5. RINCIAN PERBAIKAN F2 PRE FLIGHT DNS HARD FAIL

```
+--------------------------------------------------------------------------+
|  KONSEP                                                                  |
+--------------------------------------------------------------------------+
|  Sebelum HEAD probe HTTP berjalan, lakukan resolve DNS sinkron via       |
|  `socket.getaddrinfo` dalam executor dengan batas waktu tiga detik.      |
|  Jika `EAI_NONAME` atau `EAI_FAIL` muncul, domain dianggap mati. Tulis   |
|  blacklist ke `agentic_knowledge.json` dengan reason `dns_dead` dan      |
|  drop seed di pass ini juga. Jika timeout atau error tak dikenal,        |
|  fail open supaya blip jaringan tidak menjatuhkan URL sehat.             |
+--------------------------------------------------------------------------+

  alur sebelum F2                       alur setelah F2

  seed                                  seed
    +-> _is_discovery_seed                +-> _is_discovery_seed
    +-> _is_serp_url                      +-> _is_serp_url
    +-> HEAD probe (40 detik)             +-> _dns_resolve (3 detik)
        +-> Chromium spawn                    +-> jika mati: blacklist + drop
            +-> 20 menit ERR_NAME_NOT          +-> jika hidup: HEAD probe
                _RESOLVED dibakar                +-> jalur normal
```

## 13. F3 GOTCHA YANG TERTANGKAP SUB AGENT

```
+--------------------------------------------------------------------------+
|  TEMUAN PENTING                                                          |
+--------------------------------------------------------------------------+
|  `QueuedChatOllama` di `crawler.tools.llm.queue` menerima kwarg          |
|  `_llm_queue_tier="vision"`. Kedua jalur builder LLM di `agent.py:122`   |
|  dan `enrich_agent.py:485` memakai tier ini.                             |
|                                                                          |
|  Tanpa preservasi kwarg saat fallback rebuild, empat worker listing      |
|  paralel ditambah N worker enrich akan stampede ke `granite4.1:8b`       |
|  bersamaan saat primary jatuh. Ollama overload, fallback ikut gagal,    |
|  semua seed cooldown `llm_degraded`. Cascade.                            |
|                                                                          |
|  MITIGASI WAJIB                                                          |
|  Opsi A. Preserve `_llm_queue_tier="vision"` di builder fallback.        |
|  Opsi B. Buat tier baru `"vision_fallback"` dengan kapasitas terpisah.   |
|                                                                          |
|  Rekomendasi: opsi A, lebih sederhana dan bandwidth Ollama untuk         |
|  granite-8b cukup karena modelnya kecil.                                 |
+--------------------------------------------------------------------------+
```
